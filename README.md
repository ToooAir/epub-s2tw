# epub-s2tw

將簡體中文的 EPUB 翻譯成適合自己「舒服閱讀」的繁體中文（專為閱讀簡中電子書的台灣讀者打造）。
本專案不只是一個單純的 Google 翻譯腳本，而是一套完整的「在地化管線（Localization Pipeline）」。它除了使用機翻解決語意問題外，更整合了 OpenCC 官方台灣異體字標準、NMT 幻覺降級機制與全書全域一致性檢查，致力於產出如同台版實體書般自然、順暢且沒有機翻怪味的閱讀體驗。

> **免責聲明**：免費模式使用 Google 未公開的非官方端點，可能隨時失效或違反服務條款，請自行評估風險。

## 為什麼不用純文字轉換 (OpenCC)？

本專案與傳統的 `opencc -c s2tw` 轉換有著本質上的架構與效果差異。這裡並非只是一個「打 API 的翻譯腳本」，而是一套兼具除錯與全域統計的在地化管線 (Localization Pipeline)：

| 核心特性 | epub-s2tw | 靜態字典轉換 (如 OpenCC) |
| --- | --- | --- |
| **翻譯引擎** | **神經機器翻譯 (Google Translate)**<br>基於整段數百字的上下文進行語意對齊轉換。能精確辨識輕小說中的俚語、自創詞與語法。 | **規則與詞庫驅動轉換**<br>以豐富的詞庫為核心，支援字元、詞彙級別、異體字與地區習慣用語（如：屏幕→螢幕）轉換，但非基於語意推論的 NMT。 |
| **後處理防護** | **教育部字典驅動防禦網 + 台灣過濾鏡**<br>以教育部國語辭典（162,887 詞頭）為唯一權威，自動生成約 33,000 條修正規則，確保 Google 輸出的非標準字形被導正為 MOE 認可詞。外加 `TWVariants` 強制洗滌成台灣現代標準字形（如裏→裡），以及 bigram 邊界保護防止短規則誤殺合法詞彙。 | **靜態標準化框架**<br>可透過 `s2twp` 等配置檔與自訂詞庫執行臺灣/香港等地的標準化，但不會主動進行跨語境的語意推理與潤飾。 |
| **全域一致性校正** | **基於 N-gram 的智慧收斂與多數決**<br>具備整本電子書的全域記憶。能自動將不同章節中被機翻不穩定轉化的名詞變體（例如：雪之下 vs 雪底下），透過演算法對齊並收斂成最通用的單一版本。 | **依賴靜態詞庫的一致性**<br>依循同一套配置與詞庫即可維持極高的規則一致性，但不具備「全書動態記憶」或偵測語意變體的能力。 |
| **幻覺抵禦與防撞** | **八段防護網與絕對真理法則**<br>當 AI 發生 51% 比例的大規模幻覺（如把「政近一行」超譯為「政近一步」），或遭遇原著雙胞胎名字陷阱時，系統具備「段落競爭對齊」與「全域合規防撞網」，精準抹除機翻幻覺的同時守護原著彩蛋。 | **專注於精確的文字置換**<br>官方明確定位其並非翻譯工具。按部就班地執行配置檔轉換，不負責處理 NMT 獨有的「AI 幻覺」與「多譯名衝突」等問題。 |
| **極限斷句容錯** | **自動碎紙重譯自救機制**<br>針對 NMT 因遇到輕小說中常見的高重複疊詞（如連續的「啊啊啊」）而導致的 API 截斷不吐字現象，系統能觸發標點符號自動降級拆分，挽救丟失的文句。 | **穩定的純本地運算**<br>完全本地端執行，速度極快、容易擴充，無須顧慮 API 截斷、網路逾時或文本過長的問題。 |


## 專案結構

```
epub-s2tw/
├── translate_epub.py     # CLI 入口
├── epub_handler.py       # EPUB 讀取、文字抽取、翻譯替換、儲存
├── translator.py         # Google Translate 包裝層（快取、批次、並行、兩種模式）
├── postprocess.py        # 後處理層：修正 Google 的繁體歧義字與已知翻譯錯誤
├── dict-revised.json.xz  # 教育部國語辭典（162,887 詞頭），後處理規則生成來源
├── STCharacters.txt      # OpenCC 單字對照表（簡→繁，含多選項）
├── TWVariants.txt        # 台灣異體字標準化對照表
├── corrections.json      # 手動維護的已知修正字典（輕小說常見錯誤補丁）
├── STPhrases.txt         # （已棄用，可刪除）
├── build_corrections.py  # （已棄用）曾用於自動建立 corrections.json
├── requirements.txt
├── .env.example
└── README.md
```

## 安裝

```bash
python3 -m venv .venv

```

## 用法

### 免費模式（不需 API Key）

```bash
# 單檔
python translate_epub.py 小说.epub --free

# 多檔（shell glob）
python translate_epub.py ./*.epub --free

# 遞迴掃描整個資料夾（自動排除 output 目錄）
python translate_epub.py -d ./books --free
```

### API 模式（Google Cloud Translation v2）

```bash
python translate_epub.py -d ./books --api-key YOUR_KEY
# 或在 .env 設定 GOOGLE_API_KEY，直接執行：
python translate_epub.py -d ./books
```

### 常用選項

| 選項 | 說明 |
|------|------|
| `-d DIR` | 遞迴掃描資料夾內所有 .epub |
| `-o DIR` | 指定輸出資料夾（預設 `./output`）|
| `--free` | 使用免費 Google 翻譯 |
| `-k KEY` | 直接傳入 API Key |
| `--no-rename` | 不把輸出檔名轉為繁體 |
| `--log-consistency` | 輸出一致性修正報告 (`_consistency.txt`) |
| `--ckip` | 啟用 CKIP albert-tiny 雙重詞界確認（需安裝 `ckip-transformers`）|
| `-v` | 顯示每個 XHTML 的處理進度 |
| `--dry-run` | 只列出會處理的檔案，不實際翻譯 |
| `--clear-cache` | 清除全域快取 `.translate_cache.json` |

### 完整範例

```bash
# 預覽會處理哪些檔案（確認不含 output）
python translate_epub.py -d . --dry-run

# 免費版翻譯整個資料夾，顯示詳細進度
python translate_epub.py -d ./books --free -v -o ./繁體輸出

# 重跑（已存在的輸出會自動跳過）
python translate_epub.py -d ./books --free
```

## 後處理層

純粹的機翻或 OpenCC 都有各自的死穴。`postprocess.py` 會自動建立防護網來修正這些翻譯或轉換錯誤：

**1. 台灣異體字標準化（TWVariants 洗滌機制）**

一般簡轉繁詞庫（如 STPhrases）會輸出極度古典的考據繁體字（例如把「才會」轉成「纔會」、「裡面」轉成「裏面」）。
系統會動態載入最高權限的 `TWVariants.txt`，並充當「淨水器」，將翻譯後的所有字元與內部字典強制洗滌成台灣現代標準字形（如 纔→才、裏→裡、麪→麵），徹底消除對岸機翻古典味。

**2. 教育部字典驅動的字形修正規則**

以教育部國語辭典（`dict-revised.json.xz`，162,887 詞頭）為唯一權威來源，搭配 `STCharacters.txt` 的字形對應，自動生成約 33,000 條修正規則。規則方向保證：`wrong`（非 MOE 詞）→ `right`（MOE 認可的台灣標準詞），不會產生反向或語意錯誤的修正。同時使用 bigram 邊界保護，防止短規則誤觸更長詞的片段（如避免「髮往」規則命中「頭髮往後」）。

**3. 詞組整體修復與自訂補丁**

如「剩餘價值」被翻成「剩馀價值」等 Google 系統性詞彙翻譯錯誤。這類專屬補丁收錄於 `corrections.json` 中，支援手動維護。

後處理採 **single-pass 最長匹配**掃描，每個位置只處理一次，避免多條規則連鎖衝突。

**Layer 1 / Layer 2 分層保護：**
- `corrections.json`（Layer 1）為人工驗證的已知錯誤，**不受 bigram 邊界保護限制**，無條件套用。
- MOE 生成規則（Layer 2）才受 bigram 保護，避免誤傷合法詞彙。

**（可選）CKIP 雙重詞界確認：**
啟用 `--ckip` 後，Layer 2 的 bigram 攔截會加入第二道驗證——以 `ckip-transformers albert-tiny` 斷詞器確認是否真的跨詞界，兩者皆同意才跳過，任一放行則套用修正，大幅降低 bigram 誤傷率。

**Bigram 攔截 Log（`--log-consistency`）：**
報告末尾會附上「Bigram 攔截紀錄」，列出匹配成功但被攔截的規則與上下文，供人工確認是否需補入 `corrections.json`。

### 更新 corrections.json

`corrections.json` 目前採**手動維護**。當翻譯結果出現系統性的詞彙錯誤，且該錯誤不屬於 MOE 字形規則能涵蓋的範圍（例如 Google 把感嘆詞「喂」誤轉成動詞用的「餵」），直接編輯 `corrections.json` 新增：

```json
{
  "corrections": {
    "餵，": "喂，" 
  }
}
```

規則格式：`"wrong": "right"`，其中 `wrong` 為 Google 實際輸出的錯誤形式，`right` 為台灣標準。key 長度須 ≥ 2 字元。

## 一致性修正

翻譯完成後，會自動掃描全書統一同一原文被翻成不同繁體形式的情況。

**典型案例：** 簡體原文「艾莉莎」在不同章節被 Google 翻成「艾莉莎」或「艾麗莎」（字形混淆），一致性修正會統計兩者出現次數，將少數派改成多數派。「雪之下」也是一樣的狀況，會翻譯成「雪底下」或「雪之下」。

**演算法與多層防護網（為輕小說量身打造）：**
1. **多模態全域掃描（涵蓋 HTML 強調標籤）**：NMT 引擎的注意力機制常被 `<em>` 或 `<strong>` 等強調標籤打斷，導致傲嬌角色的粗體字發言成為名詞不一致的重災區。系統會無視標籤阻斷，將純文本與格式化文字一併納入一致性分析。
2. **高頻實體動態豁免（解決三字女主角痛點）**：為了避免 NMT 將「情緒」翻為「情感」時遭到誤判，演算法預設僅針對 4 字以上的詞彙進行修正。但考量到 ACG 作品中常有「雪之下」等 3 字核心名詞，系統導入了動態頻率掃描：當特定短名詞在全書的出現頻率達到極值時，將自動豁免長度限制，賦予其一致性正名權限。
3. **幻覺容忍閾值自適應 (`max_minor_ratio` 放寬)**：Google 翻譯有時會產生嚴重的字面幻覺（如將「雪之下」大機率翻成「雪底下」）。為了應對這類出錯率極高的死角，系統將候選字異常閾值放寬至 `33%`。只要正確譯名穩定佔據全書三分之二的比例，系統便會將剩餘的三分之一視為機器幻覺，並納入全域修正。
4. **同音異形競爭對齊（妥善處理原著字形梗）**：作者常為了區別暱稱而混用「艾莉娅」與「艾莉莎」。當 NMT 引擎因同音將其混淆為「艾莉亞」時，系統會動態掃描該段落的所有 N-gram 進行字元距離計算。若發現編輯距離存在平手的歧義狀態，系統將觸發「競爭放棄 (Competitive Abstain)」，避開傳統靜態字典的盲點，妥善保留作者刻意安排的文字彩蛋。
5. **核心精簡與防呆退回鎖 (Safe Core Rollback)**：當確立將「雪底下不」修正為「雪之下不」後，系統會提煉核心關鍵字「雪底→雪之」進行全域替換。但若核心關鍵字過短（如只有兩字），極易引發誤殺（如將「東西埋在雪底」變為「雪之」）。此時系統會觸發安全鎖，退回使用完整的 4 字元穩定字串替換，迴避短字元造成的全域波及。
6. **長短規則衝突消除 (Conflict Resolution)**：NMT 偶爾會因局部文法（如加上「的」字）產生翻譯偏移。導致長規則「雪之下的→雪底下的」試圖覆蓋全域的短規則「雪底下→雪之下」。這會造成程式迴圈或結果互相覆蓋。系統藉由分析目標詞彙互相包含的特徵，強制廢除受機率干擾的冗長規則，確保全網遵循最穩定的核心短語意。
7. **絕對真理法則（抵禦 NMT 51% 幻覺攻擊）**：純粹的多數決演算法存在盲點：若 NMT 發生大規模幻覺，高頻率將「政近一行」錯譯為「政近一步」，演算法反倒會將錯字當作多數正解。系統為此導入「繁簡等價性判定」：當原文字元全無簡繁歧義時（如「政近一行」、「艾莉莎」），其繁體對映必定為自身。系統將強制推翻 AI 的錯誤多數決結果，直接將絕對正確的原文字串冊封為最終選項。
8. **兩階段全域防撞網（迴避命名衝突的無差別抹除）**：若全域替換命中了另一名合法角色的名字時（例如將錯翻的「常陸院馨」全部修正回「常陸院光」），將導致真弟弟的名字在全書中被徹底蒸發。系統在執行替換前，會執行 `Pass 1` 全書白名單普查。若發現修復目標早已登錄於「全域合法名詞」中，將立刻攔截並終止該項替換。系統以此容許局部的翻譯失誤，嚴格換取免於損毀全書核心人物庫的安全性。

### 除錯與回報問題

若發現翻譯結果中有不自然的名詞替換，或希望確認系統進行了哪些一致性修正：

1. 使用 `--log-consistency` 重新執行或處理該 EPUB 檔案。
2. 開啟 `output/` 目錄下的 `[檔名]_consistency.txt` 查看修改紀錄。
3. 若確認為系統的越界誤判或語意破壞，歡迎至 GitHub 建立 Issue。
4. **建立 Issue 時的建議**：請一併附上該篇報告中對應錯誤的`[檔名]_consistency.txt`，這將大幅加快我們除錯與調整演算法的速度！

## 快取機制

翻譯結果儲存在 `.translate_cache.json`。同一段文字只會送 API 一次，重跑或中斷後繼續都能省下費用與時間。批次處理多本書時，跨書的重複段落（角色名、口頭禪等）同樣只翻一次。

## 翻譯引擎

免費模式使用 `translate.googleapis.com/translate_a/single` 非官方端點：

- 單次請求上限約 1800 字元，自動切塊，多段合併並行請求大幅提速。
- **神經網路幻覺防護 (NMT Fallback)**：當遇到角色語調激動（包含大量重複字眼如「我知道...我知道...我知道！」）時，Google 翻譯極易陷入迴圈，發生造詞幻覺或直接吃句（截斷）。
- **異常偵測與閃電隔離重譯 (碎紙機狙擊槍)**：一旦偵測到文字長度異常流失 (>15%) 或是句尾標點消失，終端機會拋出警告，並觸發自救機制：將該段落物理切碎，塞入 `⚡` 分隔符號強制切斷 NMT 注意力機制後，以單一請求一次性發送。這不僅完美繞過漏句 Bug，更免去了 HTTP 500 擋刷限流，達成極致的重譯效能提升。
  - **效能實測對比**：針對十五本輕小說（約 4.2 萬個段落，共 250 萬翻譯字元）進行實測。相較於傳統重譯機制，新版演算法將**總耗時自 12 分 40 秒縮減至 8 分 58 秒（提速近 30%）**；HTTP 請求總數更是從 2302 次降至 **1891 次**（平均每次請求處理高達 22.3 個段落）。若清除本地快取再次執行，受惠於 Google 遠端 NMT 本身的快取命中，整體時間更能進一步**砍半至 4 分 41 秒**。期間遭遇的 70 次 NMT 嚴重漏句與幻覺，皆被閃電隔離完美捕捉修復，全程零資料流失。
- 失敗自動以指數退讓無限重試，絕不產出夾雜簡體的半成品。

## 處理範圍

| 內容 | 說明 |
|------|------|
| 內文 XHTML | block-level 元素（p, h1～h6, li 等）全數翻譯 |
| 含 inline 標籤 | `<em>`, `<strong>` 等以 HTML 模式翻譯，保留格式 |
| nav.xhtml 目錄 | 透過 `book.toc` 更新，正確反映在電子書目錄 |
| toc.ncx | `<navLabel>` 與 `<docTitle>` 全數翻譯 |
| OPF metadata | title、description、publisher、subject、belongs-to-collection、file-as |
| 不翻譯 | `script`, `style`, `code`, `pre`, `rt`（注音）內的文字 |

## 已知限制

- 免費模式對超長段落會自動切塊，但極端情況仍可能觸發 rate limit（會自動重試）
- 內文中的注音（ruby/rt）標籤不會被翻譯
- 一致性修正目前不覆蓋 toc.ncx 與 nav.xhtml，目錄中的人名可能仍有不一致
- 若 Google 回傳的 HTML 有 entity 雙重編碼問題，可在 `epub_handler.py` 中把 `fmt="html"` 改為 `fmt="text"`
