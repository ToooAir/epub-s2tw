# epub-s2tw

將簡體中文的 EPUB 翻譯成適合自己「舒服閱讀」的繁體中文（專為閱讀簡中電子書的台灣讀者打造）。
本專案是一套完整的「在地化管線（Localization Pipeline）」，支援兩種翻譯引擎：

- **繁化姬**（推薦）：以台灣標準詞庫進行確定性字詞轉換，無 NMT 幻覺、無需 API Key，搭配全書一致性修正使用。
- **Google NMT**：神經機器翻譯，具備語意理解能力，適用於需要跨語意轉換的特殊場景。

> **免責聲明**：Google NMT 免費模式使用 Google 未公開的非官方端點，可能隨時失效或違反服務條款，請自行評估風險。

## 為什麼不用純文字轉換 (OpenCC)？

本專案與傳統的 `opencc -c s2tw` 轉換有著本質上的架構差異。除了使用比 OpenCC 更精準的轉換引擎外，更在上層建立了全書統計、後處理修正與 EPUB 管線，致力於產出無機翻怪味的閱讀體驗：

| 核心特性 | epub-s2tw | 靜態字典轉換 (如 OpenCC) |
| --- | --- | --- |
| **轉換引擎** | **繁化姬（預設）或 Google NMT 可選**<br>繁化姬以七套台灣標準字典疊加進行確定性字詞轉換，不會出現 OpenCC 的過貪婪匹配問題（如「的士氣」不會被誤觸為「計程車氣」）。Google NMT 模式另提供語意推理能力，適用於特殊場景。 | **規則與詞庫驅動轉換**<br>以詞庫為核心支援字元、詞彙級別與地區用語轉換（如屏幕→螢幕），但詞條最長匹配機制有過貪婪風險，且轉換品質上限受限於詞庫收錄完整度。 |
| **全域一致性校正** | **基於 N-gram 的智慧收斂與多數決**<br>具備整本電子書的全域記憶。能自動將不同章節中被引擎不穩定轉化的名詞變體（例如：雪之下 vs 雪底下），透過演算法對齊並收斂成全書統一的單一版本。 | **依賴靜態詞庫的一致性**<br>依循同一套配置即可維持極高的規則一致性，但不具備「全書動態記憶」或跨章節偵測語意變體的能力。 |
| **後處理修正層** | **教育部字典驅動防禦網 + 台灣過濾鏡**<br>以教育部國語辭典（162,887 詞頭）為唯一權威，自動生成約 33,000 條修正規則。`TWVariants` 強制洗滌成台灣現代標準字形（如裏→裡），bigram 邊界保護防止短規則誤殺合法詞彙。手動維護的 `corrections.json` 補丁層覆蓋引擎系統性錯誤。 | **靜態標準化框架**<br>可透過 `s2twp` 等配置檔執行台灣標準化，但規則固定，無法針對不同轉換引擎的輸出特性動態調整。 |
| **譯前實體保護** | **動態斷層偵測與雙重濾網保護**<br>利用全書詞頻統計與教育部國語辭典作為雙重濾網，精準抓出高頻人名與專有名詞，以私有標籤物理隔離後再送轉換，防止引擎因跨詞錯誤破壞名字。 | **依賴事後修補**<br>無譯前保護概念，名詞被破壞後只能人工建立靜態名單事後修補。 |
| **NMT 幻覺防護** | **八段防護網（Google NMT 模式專用）**<br>碎紙機重譯、段落競爭對齊、全域防撞網，抵禦 Google NMT 的截斷與幻覺。繁化姬模式轉換具確定性，不存在幻覺問題，此層自動停用。 | **不適用**<br>純本地端確定性運算，無 NMT 幻覺風險，但也無需此層防護。 |


## 已知重疊與局限

閱讀比較表前，有幾點值得誠實說明：

**資料來源說明**

本專案使用的字典檔案來自公開資料集：
- `STCharacters.txt`、`TWVariants.txt`：直接取自 [OpenCC](https://github.com/BYVoid/OpenCC) 專案
- `dict-revised.json.xz`：來自[教育部重編國語辭典修訂本](https://dict.revised.moe.edu.tw/)公開資料

**後處理層與 OpenCC 的重疊**

`TWVariants` 洗滌與字形修正規則，在「繁體字形標準化」這件事上與 OpenCC `s2twp` 有顯著重疊——資料來源本來就是同一份。本專案沒有直接呼叫 OpenCC 的原因是：規則生成方向不同（以 MOE 詞頭為 ground truth，反向推導 NMT 輸出的錯誤形式），以及需要 bigram 邊界保護來防止短規則誤傷。

如果你只需要純字形轉換且不在意詞彙精準度，直接用 `opencc -c s2twp.json` 更輕量。但繁化姬的詞庫品質與誤傷率均優於 OpenCC，且本專案提供完整 EPUB 管線，兩者已不在同一比較基準上。

真正無法被 OpenCC 取代的部分——全書一致性修正、譯前實體保護、後處理修正層、完整 EPUB 結構處理（toc.ncx、nav.xhtml、inline HTML）——是這套管線的核心價值，不在 OpenCC 的設計範疇內。

**EPUB 管線的既有工具**

EPUB 拆解→翻譯→重組的骨架，社群已有 [bilingual_book_maker](https://github.com/yihong0618/bilingual_book_maker) 等工具。本專案分開實作 `epub_handler.py` 的直接原因是 ebooklib（底層 EPUB 解析庫）對 `toc.ncx` 與 `nav.xhtml` 的翻譯支援不可靠——`nav.xhtml` 看起來有機會被處理，但 `toc.ncx`／書籤目錄在實務上常常沒有被正確翻譯，導致電子書閱讀器的章節列表停留在簡體。因此針對這兩個部分獨立實作了 `_translate_ncx()` 與 `_translate_toc()`。其他差異（inline HTML 保護、entity sentinel、後翻譯修復管線）則是在此基礎上逐步補上的。

**快取機制**

`.translate_cache.json` 的 MD5 快取是翻譯腳本的標準基礎功能，無創新之處。

## 專案結構

```
epub-s2tw/
├── translate_epub.py        # CLI 入口
├── epub_handler.py          # EPUB 讀取、文字抽取、翻譯替換、儲存
├── translator.py            # Google Translate 包裝層（快取、批次、並行、兩種模式）
├── zhconvert_translator.py  # 繁化姬包裝層（zhconvert.org），與 translator.py 介面相容
├── postprocess.py           # 後處理層：修正 Google 的繁體歧義字與已知翻譯錯誤
├── dict-revised.json.xz     # 教育部國語辭典（162,887 詞頭），後處理規則生成來源
├── STCharacters.txt         # OpenCC 單字對照表（簡→繁，含多選項）
├── TWVariants.txt           # 台灣異體字標準化對照表
├── corrections.json         # 手動維護的已知修正字典（輕小說常見錯誤補丁）
├── STPhrases.txt            # （已棄用，可刪除）
├── build_corrections.py     # （已棄用）曾用於自動建立 corrections.json
├── requirements.txt
├── .env.example
└── README.md
```

## 安裝

```bash
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt

# （可選）建立 .env
cp .env.example .env
# 編輯 .env 填入 GOOGLE_API_KEY
```

## 用法

### 繁化姬模式（推薦，不需 API Key）

```bash
# 單檔
python translate_epub.py 小说.epub --zhconvert

# 多檔（shell glob）
python translate_epub.py ./*.epub --zhconvert

# 遞迴掃描整個資料夾
python translate_epub.py -d ./books --zhconvert -o ./output -v
```

繁化姬以台灣標準詞庫進行字詞轉換，涵蓋國家教育研究院、教育部國語辭典、萌典等多套字典，轉換準確率高且無 NMT 幻覺問題，不需申請任何 API Key。

### Google NMT 免費模式

```bash
python translate_epub.py 小说.epub --free
python translate_epub.py -d ./books --free
```

### Google NMT API 模式（Google Cloud Translation v2）

```bash
python translate_epub.py -d ./books --api-key YOUR_KEY
# 或在 .env 設定 GOOGLE_API_KEY，直接執行：
python translate_epub.py -d ./books
```

### 常用選項

| 選項 | 說明 |
|------|------|
| `--zhconvert` | 使用繁化姬 API（zhconvert.org），無需 API Key（推薦）|
| `--free` | 使用免費 Google 翻譯 |
| `-k KEY` | 直接傳入 Google API Key |
| `-d DIR` | 遞迴掃描資料夾內所有 .epub |
| `-o DIR` | 指定輸出資料夾（預設 `./output`）|
| `--no-rename` | 不把輸出檔名轉為繁體 |
| `--no-protect-entities` | 停用譯前高頻人名實體保護機制（預設為啟用）|
| `--log-consistency` | 輸出一致性修正報告 (`_consistency.txt`)|
| `--ckip` | 啟用 CKIP albert-tiny 雙重詞界確認（需安裝 `ckip-transformers`）|
| `-v` | 顯示每個 XHTML 的處理進度 |
| `--dry-run` | 只列出會處理的檔案，不實際翻譯 |
| `--clear-cache` | 清除全域快取 `.translate_cache.json`（僅 Google NMT 模式使用）|

### 完整範例

```bash
# 預覽會處理哪些檔案（確認不含 output）
python translate_epub.py -d ./books --dry-run

# 繁化姬翻譯整個資料夾，顯示詳細進度
python translate_epub.py -d ./books --zhconvert -v -o ./output

# Google 免費版，含一致性修正報告
python translate_epub.py -d ./books --free -o ./output --log-consistency

# Google 免費版 + CKIP 雙重詞界確認
python translate_epub.py -d ./books --free -o ./output --log-consistency --ckip
```

## 後處理層

純粹的機翻或 OpenCC 都有各自的死穴。`postprocess.py` 會自動建立防護網來修正這些翻譯或轉換錯誤：

**1. 台灣異體字標準化（TWVariants 洗滌機制）**

一般簡轉繁詞庫（如 STPhrases）會輸出極度古典的考據繁體字（例如把「才會」轉成「纔會」、「裡面」轉成「裏面」）。
系統會動態載入最高權限的 `TWVariants.txt`，並充當「淨水器」，將翻譯後的所有字元與內部字典強制洗滌成台灣現代標準字形（如 纔→才、裏→裡、麪→麵），徹底消除對岸機翻古典味。

**2. 教育部字典驅動的字形修正規則** *(Google NMT 模式下效果顯著；繁化姬模式觸發率接近零)*

以教育部國語辭典（`dict-revised.json.xz`，162,887 詞頭）為唯一權威來源，搭配 `STCharacters.txt` 的字形對應，自動生成約 33,000 條修正規則。規則方向保證：`wrong`（非 MOE 詞）→ `right`（MOE 認可的台灣標準詞），不會產生反向或語意錯誤的修正。同時使用 bigram 邊界保護，防止短規則誤觸更長詞的片段（如避免「髮往」規則命中「頭髮往後」）。

**3. 詞組整體修復與自訂補丁** *(corrections.json 目前以 Google NMT 系統性錯誤為主要維護對象)*

如「剩餘價值」被翻成「剩馀價值」等系統性詞彙錯誤。這類專屬補丁收錄於 `corrections.json` 中，支援手動維護。

**4. 省略號正規化** *(Google NMT 模式專用)*

Google 翻譯有時會把台灣標準的雙省略號 `……` 壓縮成單省略號 `…`。後處理層在每段文字的最後一步統一還原為 `……`。繁化姬不會產生此問題。

**5. 固定字詞截斷／置換修復** *(Google NMT 模式專用，繁化姬模式下自動跳過)*

Google NMT 對「簡繁相同的固定字詞」（如成語 `老神在在`、疊詞 `扭扭捏捏`）有時會產生兩種系統性破壞：

| 類型 | 原文 | NMT 輸出 | 錯誤機制 |
|---|---|---|---|
| **截斷** | `老神在在地坐著` | `老神在地坐著` | 最後一字被刪除 |
| **置換** | `笑容可掬地回答` | `笑容可搬地回答` | 最後一字被替換成錯誤字 |

翻譯完成後，`_source_guided_repair_pass` 利用全書已收集的「原文↔譯文」對，自動偵測並修復這兩類錯誤。只考慮長度 ≥ 4 的固定字 MOE 詞頭，並設有三層假陽性過濾，防止把合法的不同譯法（如 `不在意`→`不在乎`）誤判為截斷。使用繁化姬時，此 pass 因 `skip_nmt_repair=True` 自動跳過。

後處理採 **single-pass 最長匹配**掃描，每個位置只處理一次，避免多條規則連鎖衝突。

**Layer 1 / Layer 2 分層保護：**
- `corrections.json`（Layer 1）為人工驗證的已知錯誤，**不受 bigram 邊界保護限制**，無條件套用。
- MOE 生成規則（Layer 2）才受 bigram 保護，避免誤傷合法詞彙。

**（可選）CKIP 雙重詞界確認：** *(Google NMT 模式下較有意義)*
啟用 `--ckip` 後，Layer 2 的 bigram 攔截會加入第二道驗證——以 `ckip-transformers albert-tiny` 斷詞器確認是否真的跨詞界，兩者皆同意才跳過，任一放行則套用修正，大幅降低 bigram 誤傷率。繁化姬模式下 Layer 2 觸發率接近零，啟用效果有限。

**Bigram 攔截 Log（`--log-consistency`）：**
報告末尾會附上「Bigram 攔截紀錄」，列出匹配成功但被攔截的規則與上下文，供人工確認是否需補入 `corrections.json`。

### 更新 corrections.json

`corrections.json` 目前採**手動維護**，以 Google NMT 的系統性輸出錯誤為主要收錄對象。當翻譯結果出現系統性的詞彙錯誤，且該錯誤不屬於 MOE 字形規則能涵蓋的範圍（例如 Google 把感嘆詞「喂」誤轉成動詞用的「餵」），直接編輯 `corrections.json` 新增：

```json
{
  "corrections": {
    "餵，": "喂，" 
  }
}
```

規則格式：`"wrong": "right"`，其中 `wrong` 為引擎實際輸出的錯誤形式，`right` 為台灣標準。key 長度須 ≥ 2 字元。

## 一致性修正

翻譯完成後，會自動掃描全書統一同一原文被轉換成不同繁體形式的情況。

**典型案例：** 簡體原文「艾莉莎」在不同章節被轉換為「艾莉莎」或「艾麗莎」（字形混淆），一致性修正會統計兩者出現次數，將少數派改成多數派。「雪之下」也是一樣的狀況，可能出現「雪底下」與「雪之下」並存的情形。

**演算法與多層防護網（為輕小說量身打造）：**
1. **多模態全域掃描（涵蓋 HTML 強調標籤）**：`<em>` 或 `<strong>` 等強調標籤會打斷引擎的處理單元，導致傲嬌角色的粗體字發言成為名詞不一致的重災區。系統會無視標籤阻斷，將純文本與格式化文字一併納入一致性分析。
2. **高頻實體動態豁免（解決三字女主角痛點）**：演算法預設僅針對 4 字以上的詞彙進行修正，避免誤判短詞的正常語意變化。但考量到 ACG 作品中常有「雪之下」等 3 字核心名詞，系統導入了動態頻率掃描：當特定短名詞在全書的出現頻率達到極值時，將自動豁免長度限制，賦予其一致性正名權限。
3. **少數派容忍閾值自適應 (`max_minor_ratio` 放寬)** *(Google NMT 模式下尤其重要)*：Google 翻譯有時會產生嚴重的字面幻覺（如將「雪之下」大機率翻成「雪底下」），造成正確形式反成少數派。系統將候選字異常閾值放寬至 `33%`，只要正確形式佔全書三分之二，即將剩餘三分之一視為錯誤並納入修正。繁化姬模式下偏差率通常遠低於此門檻，但此保護仍保持啟用。
4. **同音異形競爭對齊（妥善處理原著字形梗）**：作者常為了區別暱稱而混用「艾莉娅」與「艾莉莎」（原著刻意以一字之差區別兩個角色）。當引擎因字形相近將其混淆時，系統會動態掃描該段落的所有 N-gram 進行字元距離計算。若發現編輯距離存在平手的歧義狀態，系統將觸發「競爭放棄 (Competitive Abstain)」，妥善保留作者刻意安排的文字彩蛋。
5. **核心精簡與防呆退回鎖 (Safe Core Rollback)**：當確立將「雪底下不」修正為「雪之下不」後，系統會提煉核心關鍵字「雪底→雪之」進行全域替換。但若核心關鍵字過短（如只有兩字），極易引發誤殺（如將「東西埋在雪底」變為「雪之」）。此時系統會觸發安全鎖，退回使用完整的 4 字元穩定字串替換，迴避短字元造成的全域波及。
6. **長短規則衝突消除 (Conflict Resolution)**：引擎偶爾會因局部文法（如加上「的」字）產生轉換偏移，導致長規則「雪之下的→雪底下的」試圖覆蓋全域短規則「雪底下→雪之下」，造成程式迴圈或結果互相覆蓋。系統藉由分析目標詞彙互相包含的特徵，強制廢除受機率干擾的冗長規則，確保全書遵循最穩定的核心短語意。
7. **絕對真理法則（抵禦多數決被污染）** *(Google NMT 模式下尤其重要)*：純粹的多數決演算法存在盲點：若 Google NMT 發生大規模幻覺（如高頻率將「政近一行」錯譯為「政近一步」），演算法反倒會將錯字當作多數正解。系統導入「繁簡等價性判定」：當原文字元全無簡繁歧義時（如「政近一行」、「艾莉莎」），其繁體對映必定為自身，強制推翻多數決結果，直接冊封原文字串為最終選項。繁化姬模式下此情況幾乎不發生，但此保護仍保持啟用。
8. **兩階段全域防撞網（迴避命名衝突的無差別抹除）**：若全域替換命中了另一名合法角色的名字時（例如將錯翻的「常陸院馨」全部修正回「常陸院光」），將導致真弟弟的名字在全書中被徹底蒸發。系統在執行替換前，會執行 `Pass 1` 全書白名單普查。若發現修復目標早已登錄於「全域合法名詞」中，將立刻攔截並終止該項替換。系統以此容許局部的轉換失誤，嚴格換取免於損毀全書核心人物庫的安全性。

### 除錯與回報問題

若發現翻譯結果中有不自然的名詞替換，或希望確認系統進行了哪些一致性修正：

1. 使用 `--log-consistency` 重新執行或處理該 EPUB 檔案。
2. 開啟 `output/` 目錄下的 `[檔名]_consistency.txt` 查看修改紀錄。
3. 若確認為系統的越界誤判或語意破壞，歡迎至 GitHub 建立 Issue。
4. **建立 Issue 時的建議**：請一併附上該篇報告中對應錯誤的`[檔名]_consistency.txt`，這將大幅加快除錯與調整演算法的速度！

> 💡 **實用技巧：建立專屬的修正詞庫**  
> 在專案初期（或剛投入翻譯新的小說系列時），強烈建議您搭配 `--log-consistency` 積極收集並檢閱 `_consistency.txt` 報告。使用 Google NMT 時，報告能幫您抓出系統性誤判並快速補入 `corrections.json`；使用繁化姬時，報告主要用於確認跨章節名詞是否已被正確統一。如果您不熟悉如何建立 PR 或懶得自己挑選，也非常歡迎直接開 Issue 並附上您的 `_consistency.txt`，我會親自幫忙過濾並更新到系統內建的詞庫中喔！

## 快取機制

翻譯結果儲存在 `.translate_cache.json`。同一段文字只會送 API 一次，重跑或中斷後繼續都能省下費用與時間。批次處理多本書時，跨書的重複段落（角色名、口頭禪等）同樣只翻一次。

## 翻譯引擎

### 引擎選擇指引

本專案最初以 Google NMT 為核心，圍繞它建立了大量的幻覺偵測、截斷修復與後處理防護。隨著使用規模擴大，逐漸發現 Google NMT 在 zh-CN→zh-TW 這個特定任務上是整個管線品質的根本天花板——幻覺、字詞截斷、系統性異體字輸出等問題，需要越來越複雜的 repair pass 才能彌補，而這些 repair pass 對繁化姬的輸出觸發率接近零。

實測驗證後決定：**對於 zh-CN→zh-TW 的日常翻譯，繁化姬是更適合的預設選擇**。理由如下：

| | 繁化姬（`--zhconvert`） | Google NMT（`--free` / `--api-key`） |
|---|---|---|
| **轉換品質** | 確定性，相同輸入永遠相同輸出 | 語意對齊，但有系統性幻覺與截斷風險 |
| **誤傷率** | 極低（詞庫精確，無 OpenCC 的過貪婪匹配） | 需 NMT repair pass 持續修補 |
| **速度** | 約 40 萬字 / 20 秒 | 受批次大小與 rate limit 限制，同規模約 5～15 分鐘 |
| **API Key** | 不需要 | 免費模式可用，穩定性較低 |
| **後處理效果** | Layer 1（corrections.json）與 Layer 3（TWVariants）正常執行；Layer 2（MOE 規則）觸發率接近零 | 後處理對修正 NMT 錯誤有顯著效果 |

**適合考慮 Google NMT 的情境（理論上少見）：**
- 原文雖標記為簡體中文，但實際包含大量非中文語境（如中英夾雜的技術文件、日文直譯的學術腳注），需要語意層翻譯而非字詞轉換
- zhconvert.org 服務中斷，需要緊急備援
- 原著刻意使用某個「只存在於簡體」的詞彙作為文字梗，希望保留 NMT 的語意判斷

對於一般輕小說、網路小說的閱讀需求，繁化姬在準確度、速度與穩定性上均優於 Google NMT。

---

### 繁化姬（`--zhconvert`）

使用 [zhconvert.org](https://zhconvert.org) 公開 API（`converter=Taiwan`），以台灣標準詞庫對每個字詞進行確定性轉換。

- 字典堆疊：goo辞書、漢字ペディア、國家教育研究院、教育部重編國語辭典修訂本、萌典、汉典、粵語審音配詞字庫
- 字詞轉換具確定性，相同輸入永遠產生相同輸出，無幻覺風險
- 最大批次 8000 字，整本書（約 40 萬字）約 30 秒內完成
- HTML 標籤安全：繁化姬只轉換漢字，`<em>`、`<strong>` 等標籤原封不動
- 啟用此模式時，NMT 專用的截斷修復 pass（`_s2t_diff_repair_pass`、`_source_guided_repair_pass`）會自動跳過

> 相較 OpenCC，繁化姬不會將「的士」誤轉為「計程車」，也不會在詞語中途截斷字詞，誤傷率遠低。

**注意事項（繁化姬服務條款）：**
- 本專案使用了[繁化姬](https://zhconvert.org)的服務（https://zhconvert.org）
- 繁化姬不保證所有轉換均正確，轉換結果僅供參考；正式文件請在轉換後親自校閱

### Google NMT（`--free` / `--api-key`）

免費模式使用 `translate.googleapis.com/translate_a/single` 非官方端點：

- 單次請求上限約 1800 字元，自動切塊，多段合併並行請求大幅提速。
- **神經網路幻覺防護 (NMT Fallback)**：當遇到角色語調激動（包含大量重複字眼如「我知道...我知道...我知道！」）時，Google 翻譯極易陷入迴圈，發生造詞幻覺或直接吃句（截斷）。
- **異常偵測與閃電隔離重譯 (碎紙機狙擊槍)**：三道防線層層把關，任一觸發即啟動重譯自救：
  - **文字長度異常流失（>15%）**：NMT 截斷句子時最直接的信號。
  - **句尾標點消失**：引號、句號、問號等消失代表句子被提前中止。
  - **靜默漏譯偵測（`_is_silent_passthrough`）**：專門對付「輸出長度不變、標點完整，但內容根本沒有翻譯」的隱形 Bug。使用簡體字元存活率演算法，對 batch 中的每個子段落獨立驗算——若譯文中超過 50% 的「必須轉換簡體字」仍原封不動，即判定為靜默漏譯並觸發重譯，避免壞的結果被快取永久固化。
  
  一旦觸發，系統將該段落物理切碎、塞入 `⚡` 分隔符強制切斷 NMT 注意力機制後，以單一請求一次性發送。這不僅完美繞過漏句 Bug，更免去了 HTTP 500 擋刷限流。
  - **效能實測對比**：針對十五本輕小說（約 4.2 萬個段落，共 250 萬翻譯字元）進行實測。相較於傳統重譯機制，新版演算法將**總耗時自 12 分 40 秒縮減至 8 分 58 秒（提速近 30%）**；HTTP 請求總數更是從 2302 次降至 **1891 次**（平均每次請求處理高達 22.3 個段落）。若清除本地快取再次執行，受惠於 Google 遠端 NMT 本身的快取命中，整體時間更能進一步**砍半至 4 分 41 秒**。期間遭遇的 70 次 NMT 嚴重漏句與幻覺，皆被閃電隔離完美捕捉修復，全程零資料流失。
- **角括號保護**：Google 翻譯會把閉合角括號 `」`（U+300D）和 `』`（U+300F）替換為西式右單引號 `'`。翻譯前會將 `「」『』` 替換為 Google 不處理的 Unicode Private Use Area 字元，收到翻譯結果後再還原，確保括號完整閉環。
- 失敗自動以指數退讓無限重試，絕不產出夾雜簡體的半成品。

## 處理範圍

| 內容 | 說明 |
|------|------|
| 內文 XHTML | block-level 元素（p, h1～h6, li 等）全數翻譯 |

## 已知限制

- 免費模式對超長段落會自動切塊，但極端情況仍可能觸發 rate limit（會自動重試）
- 內文中的注音（ruby/rt）標籤不會被翻譯
- 一致性修正不覆蓋 nav.xhtml（章節標題出現頻率為 1，不觸發統計修正）；toc.ncx 已覆蓋
- 若 Google 回傳的 HTML 有 entity 雙重編碼問題，可在 `epub_handler.py` 中把 `fmt="html"` 改為 `fmt="text"`
